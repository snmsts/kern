# Kern — 多言語組版エンジン 設計メモ

2026-07-20 起稿。2026-07-21 に名前を **Kern** に確定
(カーニング = 活字間のアキの調整。box/glue のアキ量計算という芯を素直に指す
内部エンジン名。人向けの上位オーサリング層は別 system として後から別名で乗せる)。
system・パッケージ名とも `kern`。呼び出しは `kern:break-paragraph` の形で、
関数名側が素直さを担うので system 名は識別性に振れる。
(初出は Quad と名付けたが、同ドメインの先客 = Matthew Butterick の Racket 製
`quad` (下記「立ち位置」参照) があり、敬意を払って Kern に改名した。)

## 何を作るのか

Common Lisp で、**日本語を印刷品質で組める**組版エンジン。ただし日本語専用にはせず、
**他言語を後から表で足せる土台**にする。出力先も PDF に固定せず、GUI から再利用できる形にする。

作らないもの: シェーピング (HarfBuzz の領分)、PDF ライブラリ (cl-pdf を使う)。

## 目標と動機 (2026-07-21 確定)

**到達目標: 「印刷しろと言われた時に、日本語でも英語でも、出力品質で一切文句の出ない
Common Lisp 製の組版エンジンを手元に持つ」。**

- **mass-market は狙わない。** 市場調査で「新しい CJK 組版エンジンへの直接需要は薄く、
  実需は Typst / Vivliostyle / InDesign / pTeX に流れる」ことは確認済み。それでよい。
  目的は職人的な自前の道具＝**個人の最高品質**であって普及やビジネスではない。
- **動機は Common Lisp の世界を豊かにすること。** だから CL で書くこと自体に価値があり、
  クリーンな box/glue+JFM エンジンを**ライブラリ/参照実装として CL 生態系に置く**のが
  現実的な価値ポジション。品質の北極星は Typst に据えるが、堀は Typst に無い CJK 品質。
- **制約: 他言語への拡張性は必ずケアした状態で進める。** 日英を当面の品質ターゲットに
  作り込むが、その作り込みが他言語を締め出す形になってはならない。「言語規則はデータ、
  行分割コードは言語非依存」の原則 (→ 層構造・命名規約の節) を品質のためでも崩さない。
  日本語の禁則を深掘りしても、それは新言語＝表を1枚足すこと、の形を保つ。

**優先順** (市場・問題調査より、実需 × 競合空白 × 既存土台への乗りやすさ):
縦書き → ルビ → (数式は最後。Typst が得意＝最悪の土俵)。
既に実装済み: 禁則 (JFM クラス対テーブル)・和欧間アキ・Knuth-Plass・PDF 出力・TTF サブセット。

## 立ち位置

既存の cl-typesetting は「欧文・印刷・PDF」というスコープの中で**完結している**。
壊れているのではなく、要求されなかったものが無いだけ。
足りないのは最適行分割・ペナルティ・Unicode 文字クラス・逆写像で、
いずれも当時の用途では不要だったもの。恨む相手ではなく設計の教材。

近いもの: SILE (Lua)、Typst (Rust)、Quad (Racket)。
いずれも「TeX の組版アルゴリズムを取り、マクロ言語はホスト言語に任せる」路線。
ただしどれも GUI 再利用は想定していない。

**今やる理由**: 分割規則が機械可読データとして揃った。
2002 年なら表を手で起こす必要があったが、今は落として読むだけ。

- UAX #14 行分割クラス — Unicode の `LineBreak.txt`。**これは本当にデータ**
- ハイフネーションパターン — TeX から流用 (cl-typesetting が既に同梱)
- Knuth & Plass, "Breaking Paragraphs into Lines" (1981)
- JLReq「日本語組版処理の要件」— W3C、日英対訳。**ただし下記の注意**

### ★訂正: 数値の出所 (2026-07-20 実地調査)

当初「JLReq があるので仕様は揃っている」と書いたが、**これは誤り**だった。

**JLReq は要件文書であって数値仕様ではない。** 具体的な数値は
JIS X 4051 (有償の日本工業規格) に委ねられており、JLReq 本文には
行送りの比率も和欧間の空き量も書かれていない。
JLReq を実装した `jlreq` LaTeX クラスの文書も同様に数値を JIS X 4051 に委ねている。

JLReq から拾えた具体物はほぼこれだけ:

> 「3行取り」= 3 × 文字の外枠 + 2 × 行間

用語の区別は明確になった:

- **行間 (line gap)** = 文字の外枠と外枠の【あいだ】の空き
- **行送り (line pitch)** = 1行から次の行への【送り】= 外枠の高さ + 行間

つまり和文の行送りは**ベースライン間距離ではなく外枠基準**で定義される。
欧文の baseline-to-baseline とはモデルが違う。

#### 実際の数値は実装から取る

pTeX の既定値 (出典: [pTeX manual](https://asciidwango.github.io/ptex/help/xkanjiskip.html)):

| | 既定値 | 意味 |
|---|---|---|
| `\xkanjiskip` (和欧間) | `.25zw plus1pt minus1pt` | **四分アキ**、±1pt 伸縮 |
| `\kanjiskip` (和文字間) | `0pt plus.4pt minus.4pt` | **自然幅 0 (ベタ組み)**、微小に伸縮 |

CSS の `text-autospace: ideograph-alpha` は同じ和欧間に対し
**1/8 em** を推奨している (出典: [MDN](https://developer.mozilla.org/en-US/docs/Web/CSS/text-autospace))。

**pTeX の 1/4 em と CSS の 1/8 em で 2 倍食い違う。**

これは困った事実ではなく**設計の裏付け**として読むべきもの。
唯一の正解が存在しないのだから、**値はコードに焼かず ruleset の表に置く**しかない。
既定は pTeX 互換 (四分アキ) にし、CSS 互換の表も選べるようにする。

`\kanjiskip` が「自然幅 0 + 微小な伸縮」なのも重要で、
cl-typesetting の `make-inter-char-glue` (`typo.lisp:213-217`) と発想が一致している。
和文の均等割りは「字間を空ける」のではなく
**「ベタ組みを基準に、必要な分だけ伸縮させる」**。

#### ★決定版の出所: `jfm-jlreq.lua`

pTeX 系の文字クラス表と空き量は**エンジンではなく JFM (和文フォントメトリック) にある**。
pTeX の分担は3層:

| 層 | 置き場所 | 中身 |
|---|---|---|
| 文字クラス + クラス間の空き | **JFM** | 文字→クラスの表、**クラス対→glue/kern の行列** |
| 禁則 | マクロ (`kinsoku.tex` 等) | `\prebreakpenalty` / `\postbreakpenalty` |
| 全体の伸縮 | エンジンパラメータ | `\kanjiskip` / `\xkanjiskip` |

★`\prebreakpenalty` (その文字の【前】で切るペナルティ) と
`\postbreakpenalty` (【後】で切るペナルティ) — pTeX は行頭禁則と行末禁則を
**分割点の左右から見た同じもの**として実装している。本設計の結論と一致。

そして **[abenori/jlreq](https://github.com/abenori/jlreq) の `jfm-jlreq.lua` が
JLReq をデータ化したもの**。コメントに JLReq の節番号 (3.1.3, 3.1.6.a 等) が引かれている。
LuaTeX-ja の JFM は **Lua スクリプト = 素の可読データ**なので、そのまま読める。

**ライセンス: 2条項 BSD** (Copyright 2017-2024, Noriyuki Abe)。
著作権表示を保持すれば派生利用可。表の出所として使える。

確定値 (`jfm-jlreq.lua`、zw = 1em を単位とする):

```lua
kanjiskip  = {0,    0.25, 0    }   -- 自然幅0、0.25em 伸び、縮み無し
xkanjiskip = {0.25, 0.25, 0.125}   -- 四分アキ、+0.25 / -0.125
```

pTeX の既定と一致（四分アキ）。**CSS の 1/8 em が外れ値**だった。

クラスは約25個: 開き括弧 / 閉じ括弧 / ハイフン / ダッシュ / 波ダッシュ /
感嘆疑問 / 中点類 / 句点 / 読点 / 繋ぎ符号 / 繰返し記号 / 長音 / 小書き仮名 /
前置記号 / 後置記号 / 和字間隔 / ひらがな / カタカナ / 半角カタカナ /
数字(半角191・三分192・四分193) / 漢字(0) / 欧文(27) / 境界(90)。

#### ★設計に効く発見 2 つ

**(1) `priority` — 詰めは段階的で、比例配分では表現できない**

```
優先順位は，第n段階を3 - nに対応させる．
  段階   1, 2, 3, 4, 5, 6
priority 2, 1, 0,-1,-2,-3
```

実際の値は `{-1,-2}` `{0,-2}` `{0,-1}` のような**2要素**（伸び用・縮み用）。
**JLReq は詰めの順序を6段階で規定している。**

これは TeX の素朴なグルー模型では表現できない。
`spread-boxes` (`layout.lisp:193-232`) は「伸縮の重みに比例して配分し、
飽和したらロック」なので、**段階ごとに使い切ってから次へ**という挙動にならない。
TeX の無限位数 (fil/fill/filll) は「無限が有限を支配する」であって別物。

→ **段階付きグルー解決器を書く必要がある。** 本調査で一番エンジン設計に効く発見。

**(2) 行の境界そのものが文字クラス (class 90 = `parbdd`, `boxbdd`)**

行頭・行末を特別扱いせず、擬似的な文字クラスとして同じクラス対表に入れている。
「行頭の開き括弧は半角」のような規則が表からそのまま落ちる。

→ `inter-char-glue` と `break-penalty-value` は**境界クラスを取れるようにする**。
特別扱いのコードを書かない。

**(3) `ratio`** — その空きが左右どちらの文字に属するかの比率 (0, 0.5, 1, 1/3)。
行頭・行末の版面端が揃うかに効く。

#### バグ 1: `:cl-pdf` 単体では PDF が書けない

`pdf.lisp:577` が `extended-ascii-p` を呼ぶが、定義は `pdf-parser.lisp:108` —

#### バグ 2: 康熙部首との衝突 — 常用漢字の 60% が豆腐になる

| 　 U+3000 | 635 | **0** | U+2003 EM SPACE |
| 本 U+672C | 3724 | 3724 | U+672C ✓ |

```lisp
;; 現状 (zpb-ttf-load.lisp:36-39 相当) — グリフ→CP の逆引き
;; 修正案 — CP→グリフ の順引き
(loop for code from 0 to #xFFFE
      for ch = (code-char code)
      when (zpb-ttf:glyph-exists-p ch fl)
        do (let ((gid (zpb-ttf:font-index (zpb-ttf:find-glyph ch fl))))
             (setf (aref c2g (* 2 code))      (code-char (ash gid -8))
                   (aref c2g (1+ (* 2 code))) (code-char (logand gid #xFF)))))
```

#### 圧縮は既定で無効 (2026-07-20 追記)

cl-pdf は zlib 圧縮の機構を持っているが `cl-pdf.asd` の既定が `:use-no-zlib`。
salza2 を使うには **`.asd` が読まれる前に** feature を立てる必要がある:

```lisp
(pushnew :use-salza2-zlib *features*)
;; feature が変わったので強制再コンパイル
(asdf:load-system :cl-pdf :force t)
(setf pdf:*compress-streams* t)   ; 既定 nil。*compress-fonts* は既定 t
```

効果 (249 文字の和文 1 ページ):

```
13,540,421  最初 (フォント丸ごと・非圧縮)
   691,715  サブセット化のみ
   439,212  + zlib 圧縮        ← 31 倍削減
```

★注意: 圧縮を有効にすると、埋め込むバイト列が
`(simple-array (unsigned-byte 8) (*))` でないと salza2 が TYPE-ERROR を出す。
非圧縮経路では素通りするので、圧縮を切っていると露見しない。

#### バグ 3 (というより欠落): サブセット化が無い

`zpb-ttf-load.lisp:113-121` が .ttf を丸ごと読んで埋め込む。

```
漢字 7 文字 + 2 行の PDF = 13,528,352 bytes (12.90 MB)
うちフォント 13,086,996 bytes = 96.7%
```

欧文フォント (数百 KB) では誰も困らなかった。日本語では実用上避けられない。
ただし独立した中規模作業 (TTF テーブル再構築) なので後回しでよい。

### cl-typesetting: モデル層は良い。行分割が違う。

#### 良い部分 — そのまま設計の参考になる

- `boxes.lisp:72-78` `soft-box` が伸び量/縮み量 (`max-expansion`/`max-compression`) と
  重み (`expansibility`/`compressibility`) を独立に持つ
- `layout.lisp:193-232` `spread-boxes` が**次元非依存の汎用グルー解決器**。
  飽和した箱をロックして余りを再配分する再帰。縦にも横にも同じコードが効く
- `typo.lisp:213-217, 267-268` **字間グルーが既に全文字の後ろに入っている**。
  均等割りは「スペースを伸ばす」ではなく「伸びるもの全部を伸ばす」実装
- `boxes.lisp:46-50` `h-mode-mixin` / `v-mode-mixin` が既に対称

**日本語で普通なら一番高くつく「字間による均等割り」は、既に出来ている。**

#### 足りない部分

- **行分割が貪欲法** (`layout.lisp:47-173` `fit-lines`)。
  active node も demerits も badness も無い。
  リポジトリ全体を `cost|penalty|demerit|badness|tolerance` で grep してヒット 1 件、
  それは `hyphen-box` の `cost` スロット (`layout.lisp:8`) で**どこからも読み書きされていない**。
  `fit-lines` の直上に作者本人のコメント `;;This would need a complete rewrite...`
- **penalty item 型が存在しない**。禁則を載せる場所が無い
- **`cut-point-p` が boolean** (`layout.lisp:17-31`)。
  総称関数なので「切れる場所」は拡張できるが「どれくらい嫌か」は言えない。
  切れるのは空白・ハイフン・`:eol` のみ → **CJK は切れず `cannot-fit-on-line` でエラー** (`layout.lisp:167`)
- **文字クラスという概念が無い** (`specials.lisp:58-60`)。
  `*white-chars*` と `*punctuation-marks* = ".;:!?,"` を `find` するだけ
- **item 生成が拡張不可**。`put-string` / `white-char-p` / `punctuation-mark-p` は全部 `defun`。
  コア 4 ファイルの `defgeneric` は 3 つだけで、全部 `boxes.lisp` の分割・アクセサ系

## 設計

### 層構造

```
テキスト
  → itemization      ラン分割 (スクリプト/書字方向/フォント/スタイルが均一な単位)
  → BiDi 解決        UAX #9
  → シェーピング      ランごと。★今回は実装しない (和文は不要)
  → item 列          箱・グルー・ペナルティ
  → 行分割           Knuth-Plass ★言語非依存
  → 均等割り          ランごとの戦略
  → レイアウト結果    位置付き箱の木
  → 描画             バックエンド
```

**原則: 言語規則はアルゴリズムを拡張しない。ペナルティとグルーにコンパイルされる。**

- 禁則 → penalty item
- 和欧間アキ → glue item
- ハイフネーション → discretionary item
- タイ語・クメール語 (分かち書き無し) → 辞書分割の結果が penalty item

行分割器はどの言語も知らないまま全部を扱う。
**層 2 が拡張可能だったら設計が失敗しているサイン。**

規則の二枚重ね:

| 層 | 中身 | 性質 |
|---|---|---|
| 合法性 | UAX #14 のクラス表 | 全言語共通のデータ |
| 重み | JLReq の禁則ペナルティ値 | 言語ごとのポリシー (データ) |
| 最適化 | Knuth-Plass | 完全に言語非依存のコード |

UAX #14 は「切れる/切れない/必ず切る」しか言わない。
「切れるが避けたい」の階調が JLReq から来る。
**新言語対応 = 表を 1 枚足すこと。コードは増えない。**

### 命名規約 — 汎用 API に言語名を入れない (2026-07-21)

層構造の思想を名前にも通す。**行分割器が言語を知らないなら、その API 名も言語を名乗ってはいけない。**

| 対象 | 規則 | 例 |
|---|---|---|
| 汎用 API (公開する入口・エンジン) | **操作名で付ける。言語は名前に入れない** | `break-paragraph` / `layout-paragraph` / `set-glue` / `char-class-of` / `compose` |
| 言語の選択 | **名前ではなく引数 (ポリシー/ruleset) で渡す** | `(kern:compose items :rules (jlreq-ruleset))` — 日本語性は ruleset データ側に住む |
| 言語特化ロジック | 特化名は OK。ただし `ja`/`jp` ではなく**規格名で precise に** | `jfm-*` / `jlreq` / `kinsoku`。「どの規格か」が名前で分かる方が強い |
| デモ・出力ファイル | ここだけ `ja` を許す (「これは日本語の例だ」と言ってよい場所) | `run-ja-pdf` / `ja-kern.pdf` |

**`typeset-japanese` のように言語を関数名へ埋めない。** 関数は総称のまま、`ja` らしさは渡すデータが持つ。
これは「言語規則は penalty/glue にコンパイルされて入ってくる」の名前版であり、崩れたら層 2 が名前空間に漏れているサイン。

### 定義するプロトコル 3 つ

```
   [フォント源]                              [バックエンド]
        |                                          ^
        | (1) メトリクス                            | (3) 描画
        v                                          |
   テキスト --(2)--> item列 --> Knuth-Plass --> レイアウト結果
              item生成                          (位置付き箱の木)
```

1. **メトリクスプロトコル** — `glyph-advance` / `ascent` / `descent` / `kerning` / `line-height`。
   実装は cl-pdf 用と GUI ツールキット用。cl-typesetting の実測 6 関数が規模の目安
2. **item 生成プロトコル** — 文字クラス表、分割規則、均等割り戦略。ランを受けて item 列を吐く。
   **ここが言語拡張点**
3. **描画プロトコル** — レイアウト結果を歩いて描く。PDF / GUI

(1) と (3) が両方あって初めて「PDF に流さず GUI で再利用」が成立する。

### 不可逆な決定 — v1 で必ず入れるもの

#### 逆写像 (source-start / source-end)

GUI で使うなら必須。「このピクセルは原文の何文字目か」「原文 N 文字目は画面のどこか」。
**すべての箱が原文位置を保持している**必要がある。

```lisp
(defclass box () (... (source-start) (source-end) ...))
```

印刷専用なら完全に無駄なスロット。だから普通は入れないし cl-typesetting にも無い。
後から入れると item 生成の全経路と行分割の分割・合成の全経路に追跡を通すことになる = 書き直し。

**コストはスロット 2 つ。入れ忘れたら不可逆。v1 に入れる。**
ただしそれを使う GUI バックエンドは v2 でよい。**穴だけ開けて塞がない。**

#### 単位

内部をポイントに固定しない。抽象単位 (有理数か定数倍整数) にしてバックエンドが DPI 換算する。

関連する既知の問題: 画面と印刷でフォントメトリクスが違う (量子化・ヒンティング) ため
同じ文書が違う行分割になりうる。既存解は「常に印刷メトリクスで組み、画面は拡大縮小のみ」
(InDesign 系の割り切り)。先に決めておく。

#### item 生成の入口は「文字列」ではなく「ラン」

`(スクリプト, フォント, グリフ列, 送り幅) → 箱` というインタフェースにする。
シェーピングは実装しないが、これだけで将来 HarfBuzz を噛ませる合流点が確保される。

#### 均等割り戦略を差し替え可能にする

欧文 = スペース伸縮、和文 = 字間伸縮 + 詰め、アラビア語 = カシーダ → スペース。
**グルー解決器は戦略を知らない。戦略がどんな glue を吐くかを決める。**

### 性能上の注意 (2026-07-20 実測により全面改訂)

**当初「和文は全文字間が breakpoint だから tolerance で刈らないと膨らむ」と書いたが、
診断が誤りだった。** 実測すると計算量を支配していたのは別の要因。

素朴に実装すると active node が入力長に線形、総計算量が二乗になる:

```
文字数   active最大    辺数        時間
   500       460      81,196     0.015
 1,004       994     377,924     0.069
 2,016     2,062   1,605,627     0.285
 8,005     8,551  25,789,029     4.530
```

原因は**候補の鍵に行番号を含めていたこと**。1行が 35〜45 字で揺れると、
段落が長くなるほど「その位置に到達しうる行数」の幅が広がり、
その分だけ active node が増える。breakpoint の多さではなく段落の長さが効く。

**行番号を鍵に含める必要があるのは次の場合だけ:**
- 行ごとに幅が変わる (回り込み)
- TeX の `\looseness` のように行数そのものを操作したい

最小 demerits だけが目的なら、行数の差は `line-penalty` が既に勘定している。
行幅が一定なら鍵は fitness だけでよい。

```
文字数   active最大    辺数      時間
   500       148     41,378    0.021
 1,004       156     99,253    0.025
 2,016       156    212,941    0.054
 4,002       156    431,270    0.095
 8,005       156    874,567    0.185     ← 24 倍速
```

**active は入力長によらず一定 (156 ≈ 行幅ぶんの寿命 40 × fitness 4)、時間は線形。**
総 demerits は 2,586 / 2,600 で改訂前と完全に一致するので、
同じ最適解に到達している。品質を落とした高速化ではない。

★残る注意: **回り込み (行ごとに幅が変わる) を実装すると二乗側に戻る。**
その時は行番号の幅に上限を設けるなどの追加の刈り込みが要る。
v1 は矩形の版面だけなので当面は問題にならない。

★tolerance は品質の閾値であって計算量の主因ではなかった。ただし刈り込み自体は必要。

差分再レイアウトは幸い自然に効く。Knuth-Plass は元々段落単位なので、
変更された段落だけ組み直せばよい。段落の識別子を保持しておく程度で済む。

### 収まらないものの扱い

箱とグルーに収まらない例と既存解:

- **アラビア語のカシーダ** — 「伸びると接続線を描く glue」としてモデル化できる。
  グルー解決器は幅しか見ないので、描画時のフックがあれば成立。
  シェーピング時に優先度付きの伸長点を出し、均等割りが消費する。
  なお実装によってはカシーダを諦めスペース伸縮で済ませているものも現役
- **インド系の複雑文字** — クラスタ整形が必要。層 3 (シェーピング) の話で組版の下
- **循環参照** — 行分割には幅が要り、幅にはシェーピングが要るが、
  シェーピング結果は分割位置で変わりうる。現実解は「一度整形 → 分割 → 分割点周辺だけ再整形」。
  理論的に閉じていない部分
- **ベースライン混在** — 和欧混植で効く。和文は中央寄り、欧文はアルファベティック。
  OpenType `BASE` テーブルかエンジン側規則。**JLReq が扱っている。カシーダより先に遭遇する**

**和文はシェーピングがほぼ不要** (縦書きの字形置換を除く)。
日本語は複雑文字体系の中では例外的に楽。今回は層 3 を丸ごと飛ばせる。

## 約物の幅モデル (2026-07-21)

JFM は括弧・中点・句読点を **width = 0.5 (字面のみ半角)** とし、
残り半角のアキを**隣接するクラス対 glue** に持たせる。
エンジンもこれに合わせた:

- glyph-box の advance は JFM の字面幅 (約物なら 0.5em)。フォントの実グリフは
  全角の枠に入っているので、box に glyph-offset を持たせ、クラスの align
  (right/middle/left) に応じて枠内の描画位置をずらす。
- 当初エンジンは全文字に全角を与えていたので、句点＋後続 glue が 1.5em を占め、
  約物が緩く見えた。字面幅に直して二重計上を解消。

★早とちりの記録: 「句点の直後に始め括弧が来たら密着すべき」と考えたが誤り。
  JFM は `[6]→[1] = {0.5, stretch_width, 0, priority={-1,0}}` を実際に持つ。
  句点(0.5) + アキ(0.5) + 括弧(0.5) で両者が各全角相当を占めるのが JLReq 準拠。
  アキは priority -1 で伸縮するので追い込み/追い出しの原資になる。
  出力の見た目のアキは仕様どおりで、バグではなかった。

★JFM のクラス27 (欧文) は `chars = {'alchar'}` という特殊トークンで、
  個別の文字を列挙しない。ASCII 図形文字範囲 (U+21..U+7E) を 27 に落とす。
  さらにラテンの連なりは1つの box にまとめる (1文字ずつだと全文字間が
  分割点になり欧単語が途中で改行される)。

## 行長調整 — 追い込み・追い出し (JIS X 4051 §4.19, 2026-07-21 精読)

品質を最も左右する急所。「文句ゼロ」の可否はほぼここで決まる。
参照規範は **JIS X 4051:2004 §4.19**。JLReq §3.8 はこれを解説する層で、
数値算法は JIS に委ねている (→ 立ち位置の「数値は JIS X 4051 に委ねられている」)。

### なぜ要るか

行頭行末そろえ (justification) では、行長がぴったりにならない行が必ず出る。原因:
- **禁則**: 句読点・括弧の行頭行末配置禁止で、行末の字数が版面幅からずれる
- **和欧混植**: 和欧間の四分アキが入り、全角グリッドが乱れる
- **字単位折り返し**: 和文は語でなく字で折り返すので指定行長と一致しない

過不足を、約物・和欧間・欧文間の**アキを段階的に詰める/空ける**ことで吸収する。

### 詰める処理 (追い込み) — 行が長すぎる時、この順で詰める

各段は「文字サイズ比で均等に」詰め、その段の下限に達したら次段へ。

| 段 | 対象のアキ | 既定 → 下限 |
|---|---|---|
| 1 | 欧文間隔 | (三分) → **四分**まで |
| 2 | 中点類の前後の四分アキ | 四分 → **ベタ(0)** まで |
| 3 | 始め括弧類の前・終わり括弧類の後ろの二分アキ | 二分 → **ベタ(0)** まで |
| 4 | 和欧間・和文と連数字・和文と単位記号の四分アキ | 四分 → **八分**まで |

- **句点類の後ろのアキは詰め対象に入っていない** (§4.19 a の列挙に無い)。全角アキ既定を保つ。
- 割注文字列は対象外。

### 空ける処理 (追い出し) — 行が短すぎる時、この順で空ける

| 段 | 対象のアキ | 既定 → 上限 |
|---|---|---|
| 1 | 欧文間隔 | → **二分**まで |
| 2 | 和欧間・連数字間・単位記号間の空き | → **二分**まで (処理系定義で三分まで可) |
| 3 | §6.1.4 の延ばし可能条件に合致した字間 | → **四分**まで |
| 4 | (1〜3で足りなければ) §6.1.3 の分割可能文字間を均等に空ける | 上限なし・最後の手段 |

- 割注を含む行は対象外。
- **伸長は詰めの厳密な逆ではない**: 欧文間は詰め下限=四分だが伸長上限=二分。和欧間も非対称。

### Kern への写像 — priority スロットに落ちる

この段階順は **glue の `shrink-priority` / `stretch-priority` そのもの**。
既に `items.lisp` が持っているスロットに、各段を優先度レベルとして割り当てる:

- **詰め**: 段1(欧文間)=最優先 shrink-priority、段2(中点)、段3(括弧二分)、段4(和欧間)=最低。
  各 glue は「既定 advance」「下限までの shrink 量」を自分のクラス対から持つ (JFM 由来テーブルが供給)。
- **空け**: 段1(欧文間)、段2(和欧間)、段3(延ばし字間)、段4(分割字間) を stretch-priority に。
- 分業は既存設計どおり: **Knuth-Plass は総量しか見ない**ので段階順の影響を受けず、
  **`set-glue` (グルー解決器) が §4.19 の優先度順に実際の量を配る**。
  つまり行長調整は「行分割の後、確定した各行に対する set-glue の仕事」。
- 必要なデータ = クラス対ごとの (既定アキ, 下限, 上限, shrink/stretch priority)。
  約物・和欧間はここまでで JFM 由来テーブルに乗っている。**欧文間隔だけは
  和文グルーでなく欧文側の space glue** なので別途持つ (段1の主役)。

★実装の第一歩はこの表を `setglue.lisp` の優先度順消費に落とすこと。
  ペナルティ由来の禁則も、和欧間アキも、既に box/glue/penalty に乗っているので、
  §4.19 は「既存グルーに正しい priority と下限/上限を与える」作業に還元される。

★実装状況 (2026-07-21):

    → 空けも段1(欧文間) > 段2(和欧間) > 段3(kanjiskip字間) の順で正しく並ぶ。
  - **未了・品質上の穴**: `set-glue` の overuse 分岐 (全段消費後に残余を全参加者へ比例配分=

    伸び、行が緩む (rivers)。本来は §4.19 **空け段4「§6.1.3 分割可能文字間を空ける」**へ
    行くべき箇所。段4 (と、その前に欧文間/和欧間を上限で止めて字間へ回す制御) が次の作り込み。
    (欧文・和欧混植の常用行はここまでで詰め全段＋空け段1-3 が効くので当面は十分。)

  やるなら #40 にコメントして「quicklisp 版も壊れている」「cl-typesetting 経由でも踏む」
  という情報を足す。diff はもう足りている
- **パッチ2（康熙部首）は投げる。** 誰も報告していない。マージは期待しないが、
  次に踏んだ人のために測定値ごと記録を残す
- **vendor が本線。** 門番がいないので「上流に通るか」でパッチを選別する必要は無い。
  上流の形を保つ理由は、上流が復活したときの自分のリベース容易性のみ
- サブセット化・ToUnicode・DW・重複 W・Identity-V・CMYK まで当てると
  **事実上 cl-pdf のメンテナになる**。そのコストは組版エンジンとは別勘定
- 公開フォークにするかは、必要なパッチが何本か見えてから判断する。
  自前 PDF ライタは今は損（オブジェクトモデル・xref・zlib・PNG・動く Type0 を捨てることになる）

## 適用済みパッチ (vendor/cl-pdf, 未コミット)

   ※ 上流 #40 は pdf.lisp へ移す案。どちらでも動く

## 再現手順

`probe/` に調査スクリプトがある。上流クローンは一時ディレクトリだったので消えている。

```bash
git clone --depth 1 https://github.com/mbattyani/cl-pdf.git
git clone --depth 1 https://github.com/mbattyani/cl-typesetting.git   # 参考用
```

`probe/*.lisp` 冒頭の `*base*` をクローン先に合わせてから:

```bash
/c/msys64/ucrt64/bin/ros.exe run -- --disable-debugger --load probe/probe2.lisp --quit
```

- `probe.lisp` — 最初の試行。バグ 1 (`extended-ascii-p`) を踏んで死ぬ
- `probe2.lisp` — バグ 1 を回避し、バグ 2 を測定し、PDF を出力
- `probe3.lisp` — バグ 2 の実害を頻出漢字で測定し、順引きによる修正を検証

環境: Roswell 26.02.116 (`/c/msys64/ucrt64/bin/ros.exe`)、sbcl-bin 2.6.4。
`ros.exe` は ucrt64 版を使うこと (CLANGARM64 版は Permission denied)。
